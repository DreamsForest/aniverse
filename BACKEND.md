# AniToki — бэкенд (аккаунты, комментарии, модерация)

Раньше сайт был полностью статическим. Теперь к нему добавлена серверная
часть на **Cloudflare Pages Functions + D1** (SQLite). Она даёт:

- регистрацию и вход с **подтверждением почты** (без подтверждения аккаунт
  не может комментировать);
- **комментарии** под каждым тайтлом (нужен подтверждённый аккаунт);
- **фильтр запрещённых слов** — мат/оскорбления не публикуются (проверка на
  сервере, обойти из браузера нельзя);
- **лайки/дизлайки** комментариев;
- **жалобы** на комментарий и на пользователя → уходят админу;
- **админку** `/admin.html`: список жалоб, **бан** и **мут** на 1–7 дней;
- меры безопасности (см. раздел «Безопасность»).

Весь код серверной части — в папке `functions/`. База описана в `schema.sql`.

---

## Что нужно один раз настроить

Понадобится бесплатный аккаунт Cloudflare и Node.js 18+.

### 1. Установить wrangler и войти

```bash
npm install -g wrangler
wrangler login
```

### 2. Создать базу D1 и прописать её ID

```bash
wrangler d1 create aniverse
```

Команда выведет `database_id = "..."`. Вставь его в `wrangler.toml`
(поле `database_id`).

### 3. Создать таблицы

```bash
# на «боевой» базе в облаке:
wrangler d1 execute aniverse --remote --file=schema.sql
# и локально (для wrangler pages dev):
wrangler d1 execute aniverse --local --file=schema.sql
```

### 4. Привязать базу к проекту Pages

Если сайт уже задеплоен как Pages-проект:
**Cloudflare Dashboard → Pages → твой проект → Settings → Functions →
D1 database bindings →** добавь биндинг с именем **`DB`** и выбери базу `aniverse`.

(Либо при деплое через `wrangler pages deploy` биндинг берётся из `wrangler.toml`.)

### 5. Задать переменные и секреты

Обычные переменные уже есть в `wrangler.toml` (`SITE_URL`, `MAIL_FROM`,
`ADMIN_EMAIL`). Поменяй `SITE_URL` на свой домен, если он другой.

**Секреты** (их НЕЛЬЗЯ держать в репозитории) задаются отдельно:

```bash
wrangler pages secret put RESEND_API_KEY      # ключ из resend.com
wrangler pages secret put ADMIN_SETUP_SECRET  # длинная случайная строка
```

Или в дашборде: **Settings → Environment variables →** добавь их как
**Secret** (тип «Encrypted»).

### 6. Настроить отправку почты (Resend)

1. Зарегистрируйся на https://resend.com, создай **API Key** → положи его в
   секрет `RESEND_API_KEY` (шаг 5).
2. Для нормальной доставки **подтверди свой домен** в Resend (добавишь DNS-записи)
   и укажи `MAIL_FROM` вида `AniToki <noreply@твойдомен>`.
3. Для быстрого старта можно временно использовать адрес `onboarding@resend.dev`
   (письма уходят только на твою почту-владельца аккаунта Resend).

Если `RESEND_API_KEY` не задан — сервер не падает: ссылку подтверждения он
печатает в лог (`wrangler pages deployment tail` или консоль `pages dev`).
Удобно для локальных тестов.

### 7. Задеплоить

- Через Git: запушь репозиторий — Cloudflare Pages соберёт проект сам
  (build command пустой, output dir — корень).
- Или вручную: `wrangler pages deploy .`

### 8. Создать аккаунт администратора

Один раз, после деплоя, вызови защищённый секретом эндпоинт. Email админа
берётся из `ADMIN_EMAIL` (сейчас `rowenhousecyqx@outlook.com`), пароль ты
задаёшь сам:

```bash
curl -X POST https://твойдомен/api/admin/setup \
  -H "Content-Type: application/json" \
  -d '{"secret":"ЗНАЧЕНИЕ_ADMIN_SETUP_SECRET","password":"твой-надёжный-пароль"}'
```

Ответ `{"ok":true,...}` — админ создан (или обновлён). Теперь зайди на сайт,
нажми **«Войти»**, введи email админа и заданный пароль. В меню аккаунта
появится **«🛡 Админка»** → страница `/admin.html` со списком жалоб и
кнопками бана/мута.

> Этот же запрос можно повторить, чтобы **сбросить пароль** админа.
> После первичной настройки секрет `ADMIN_SETUP_SECRET` можно сменить/удалить.

## Локальная разработка

```bash
# применить схему локально (один раз)
wrangler d1 execute aniverse --local --file=schema.sql
# скопировать переменные
cp .dev.vars.example .dev.vars   # и отредактировать
# запустить сайт + функции локально
wrangler pages dev .
```

Откроется http://localhost:8788 — там работают и статика, и `/api/*`.

> Просто открыть `index.html` файлом (`file://`) для теста аккаунтов нельзя —
> API доступен только когда сайт отдаётся сервером. Без сервера сайт работает
> как раньше (каталог/плеер), а блок комментариев показывает подсказку.

## Как это устроено (коротко)

```
functions/
├── api/
│   ├── _middleware.js         — CSRF-проверка Origin + перехват ошибок
│   ├── auth/                  — register, verify, login, logout, me, resend
│   ├── comments/[animeId].js  — список (GET) и создание (POST) комментариев
│   ├── comment/[id]/          — vote (лайк/дизлайк), report (жалоба), DELETE
│   ├── report-user.js         — жалоба на пользователя
│   └── admin/                 — reports (жалобы), moderate (бан/мут), setup
└── _lib/                      — общие модули (auth, badwords, email, …)
```

Клиент: `auth.js` (кнопка аккаунта + модалка входа/регистрации),
`comments.js` (виджет комментариев), `admin.html` (панель модерации).
Блок комментариев монтируется в `<div id="comments-root" data-anime-id="…">`
на странице тайтла (и в SPA, и на статических `/anime/<id>/`).

### Запрещённые слова

Список и логика — в `functions/_lib/badwords.js`. Можно дополнять массив
`ROOTS` (корни слов) и `ALLOW` (исключения от ложных срабатываний).
Проверка нормализует текст (латиница/цифры-двойники → кириллица, убирает
разделители), поэтому ловит обходы вроде `с у к а` и `c y k a`.

## Безопасность — что уже сделано

- **Пароли** хранятся как PBKDF2-HMAC-SHA256 (210k итераций) со случайной
  солью у каждого пользователя. Чистый пароль нигде не хранится.
- **Сессии:** в куке — случайный токен, в базе — только его SHA-256.
  Кука `HttpOnly`, `Secure`, `SameSite=Lax` (JS её не прочитает, по HTTPS,
  не уходит на сторонние сайты).
- **SQL-инъекции исключены:** все запросы — параметризованные (prepared
  statements D1).
- **XSS:** пользовательский текст экранируется при выводе на клиенте.
- **CSRF:** `SameSite=Lax` + серверная проверка заголовка `Origin` на всех
  изменяющих запросах.
- **Подбор/перечисление:** единые сообщения об ошибке входа; на регистрацию
  существующего email отвечаем нейтрально (не раскрываем, что он занят).
- **Антиспам:** ограничители частоты на регистрацию, вход, повторную отправку
  письма и комментарии (не чаще 1 за 15 сек, 20 за 10 мин).
- **Доступ:** все `/api/admin/*` проверяют роль `admin` по сессии; админа
  нельзя забанить/замутить; бан выкидывает пользователя из всех сессий.
- **Секреты** (ключ Resend, секрет инициализации) — только в зашифрованных
  переменных Cloudflare, не в коде.

### Что стоит докрутить позже (по желанию)

- Капча на регистрацию (Cloudflare Turnstile) — против ботов.
- Сброс пароля по почте (сейчас есть только подтверждение).
- Пагинация комментариев при большом объёме (сейчас отдаётся до 200).
- Перенести антиспам на KV/Durable Objects при высокой нагрузке.
